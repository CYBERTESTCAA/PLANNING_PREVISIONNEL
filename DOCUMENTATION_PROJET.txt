1. VUE D'ENSEMBLE
─────────────────
Nom du projet  : ShiftWise Planner
Type           : Application web interne de gestion de planning d'ateliers
Objectif       : Planifier les affectations d'employés sur des chantiers,
                 gérer les absences, suivre l'avancement des commandes,
                 synchroniser les données depuis Microsoft Fabric (ERP cloud).
Architecture   : Monorepo npm workspaces — 2 projets séparés
  Frontend     : React (port 8080)
  Backend      : API REST Node.js (port 3001)
  Base de données : SQLite locale (fichier dev.db dans api/)

4. BASE DE DONNÉES SQLITE — SCHÉMA COMPLET
──────────────────────────────────────────

Fichier    : api/dev.db
ORM        : Prisma v6
Config     : api/.env → DATABASE_URL=file:./dev.db
Init SQLite: PRAGMA journal_mode=WAL; PRAGMA busy_timeout=5000;
             PRAGMA synchronous=NORMAL; PRAGMA foreign_keys=ON;

ÉNUMÉRATIONS (Prisma Enums → SQLite TEXT)
  AbsenceType  : CP | RTT | MALADIE | FORMATION | AUTRE
  SlotType     : AM | PM | FULL
  ProjectStatus: A_PLANIFIER | EN_COURS | BLOQUE | TERMINE
  TaskStatus   : PENDING | DONE | NOT_DONE

─── TABLE Subsidiary (Filiale d'entreprise) ─────────────────────────────────
  id         String   PK uuid()
  code       String   UNIQUE — ex: "CAA", "STB"
  name       String   — ex: "CAA GUADELOUPE"
  createdAt  DateTime auto
  updatedAt  DateTime auto
  Relations  : → 1:N Workshop

─── TABLE Workshop (Atelier) ────────────────────────────────────────────────
  id           String   PK uuid()
  subsidiaryId String   FK→Subsidiary (CASCADE DELETE)
  code         String   — ex: "PRINCIPAL", "CARROSSERIE", "MECANIQUE"
  name         String   — ex: "Atelier CAA GUADELOUPE"
  themeColor   String?  — couleur HSL ex: "217 91% 50%"
  createdAt    DateTime auto
  updatedAt    DateTime auto
  UNIQUE(subsidiaryId, code)
  Relations    : → 1:N Team, Employee, Project
  NOTE: Unité principale de planification. Un atelier appartient à une filiale.
        Plusieurs ateliers possibles par filiale.

─── TABLE Team (Équipe — maintenu en BDD, non actif dans l'UI actuelle) ─────
  id         String   PK uuid()
  workshopId String   FK→Workshop (CASCADE DELETE)
  name       String
  isActive   Boolean  default true
  UNIQUE(workshopId, name)
  Relations  : → 1:N Employee
  NOTE: Sous-groupe d'un atelier. Présent en BDD mais l'UI a migré vers les
        workshops comme unité principale.

─── TABLE Employee (Employé / Salarié) ──────────────────────────────────────
  id         String   PK uuid()
  workshopId String   FK→Workshop (CASCADE DELETE) ← atelier de rattachement
  teamId     String?  FK→Team (SET NULL on delete) ← sous-groupe optionnel
  code       String   — code salarié ERP
  lastName   String
  firstName  String
  isActive   Boolean  default true
  UNIQUE(workshopId, code)
  Relations  : → 1:N Assignment, Absence, Task

─── TABLE Project (Chantier / Commande client) ──────────────────────────────
  id            String        PK uuid()
  workshopId    String        FK→Workshop (CASCADE DELETE)
  code          String        — code commande ERP
  label         String        — titre du chantier
  color         String        — couleur HSL déterministe (hachage du code)
  contractStart DateTime?     — date début contractuelle (source ERP)
  contractEnd   DateTime?     — date livraison contractuelle (source ERP)
  plannedStart  DateTime?     — début prévisionnel (saisi manuellement)
  plannedEnd    DateTime?     — fin prévisionnelle (saisie manuellement)
  status        ProjectStatus default A_PLANIFIER
  progressPct   Int           default 0 (0 à 100)
  isActive      Boolean       default true
  UNIQUE(workshopId, code)
  Relations     : → 1:N ManufacturingOrder, Assignment, Task

─── TABLE ManufacturingOrder (Ordre de Fabrication — OF) ────────────────────
  id        String  PK uuid()
  projectId String  FK→Project (CASCADE DELETE)
  code      String  — code OF ERP
  label     String? — désignation article
  UNIQUE(projectId, code)
  Relations : → 1:N Assignment

─── TABLE Assignment (Affectation employé-chantier) ─────────────────────────
  id                   String   PK uuid()
  employeeId           String   FK→Employee (CASCADE DELETE)
  projectId            String   FK→Project (CASCADE DELETE)
  manufacturingOrderId String?  FK→ManufacturingOrder (SET NULL)
  date                 DateTime — date de l'affectation
  slot                 SlotType default FULL — AM / PM / FULL
  timeSpentMinutes     Int?     — temps passé en minutes (optionnel)
  comment              String?  — commentaire libre
  INDEX(employeeId, date)
  INDEX(projectId)
  RÈGLE: Un employé peut avoir AM + PM le même jour = 2 lignes.
         Un employé peut avoir plusieurs chantiers sur le même slot.

─── TABLE Absence ────────────────────────────────────────────────────────────
  id         String      PK uuid()
  employeeId String      FK→Employee (CASCADE DELETE)
  date       DateTime    — jour d'absence
  type       AbsenceType — CP / RTT / MALADIE / FORMATION / AUTRE
  comment    String?
  UNIQUE(employeeId, date) ← une seule absence par jour par personne

─── TABLE Task (Tâche assignée) ──────────────────────────────────────────────
  id              String     PK uuid()
  employeeId      String     FK→Employee (CASCADE DELETE) — destinataire
  projectId       String?    FK→Project (SET NULL) — lien chantier optionnel
  title           String
  description     String?
  dueDate         DateTime?  — date d'échéance
  status          TaskStatus default PENDING
  responseComment String?    — commentaire de réponse de l'employé
  respondedAt     DateTime?  — horodatage de la réponse
  INDEX(employeeId), INDEX(projectId)

6. CONNEXION MICROSOFT FABRIC — SYNCHRONISATION ERP
─────────────────────────────────────────────────────

Microsoft Fabric est le data warehouse cloud de l'entreprise, connecté à l'ERP.
La synchronisation lit les tables ERP et les importe dans SQLite en local.

VARIABLES D'ENVIRONNEMENT REQUISES (api/.env)
  DATABASE_URL=file:./dev.db
  FABRIC_TENANT_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
  FABRIC_CLIENT_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
  FABRIC_CLIENT_SECRET=votre-secret-azure-ad
  FABRIC_SQL_ENDPOINT=xxxxxxxx-xxxx.datawarehouse.fabric.microsoft.com
  FABRIC_DATABASE=nom_du_warehouse_fabric

AUTHENTIFICATION AZURE AD
  Librairie : @azure/identity → ClientSecretCredential
  Scope     : https://analysis.windows.net/powerbi/api/.default
  Mécanisme : Le Service Principal obtient un token OAuth2 Azure AD
              avant chaque connexion. Ce token est passé à mssql via
              type: 'azure-active-directory-access-token'.
  Chiffrement: encrypt=true, trustServerCertificate=false

TABLES ERP LUES (T-SQL vers Fabric SQL Endpoint)

  1. [Entreprise]  →  Subsidiary + Workshop "PRINCIPAL"
     SELECT [IdEntreprise], [Entreprise]
     Pour chaque entreprise : upsert Subsidiary(code, name)
     + upsert Workshop(code='PRINCIPAL', name='Atelier {name}')

  2. [Salarié]  →  Employee
     SELECT [IdSalarié], [Nom salarié], [Prénom salarié],
            [Code salarié], [Code entreprise]
     Associe chaque salarié au workshop de son entreprise.
     Upsert sur (workshopId, code).

  3. [Commande client ligne]  →  Project
     SELECT [Code commande], [Titre commande], [Code et titre commande],
            [Date commande], [Date livraison], [Etat avancement], [IdEntreprise]
     Déduplication par [Code commande] (table multi-lignes par article).
     contractStart ← [Date commande]
     contractEnd   ← [Date livraison]
     status        ← mappage [Etat avancement]

  4. [Dépense temps passé]  →  ManufacturingOrder (OF)
     SELECT DISTINCT [Code OF], [Code commande client], [IdArticle]
     Lie chaque OF au Project correspondant.

MAPPING STATUT ERP → STATUT INTERNE
  Code ERP 0, 1  →  A_PLANIFIER
  Code ERP 2, 3  →  EN_COURS
  Code ERP 4     →  BLOQUE
  Code ERP 5, 6  →  TERMINE
  (Codes non reconnus → EN_COURS par défaut)

COULEURS DES CHANTIERS
  Algorithme golden-angle hue sur le code commande (hash 16 bits).
  Résultat HSL stable et unique : même code = même couleur entre les syncs.
  Format : "{hue} {sat}% {light}%" — ex: "184 72% 48%"

FRÉQUENCE
  Automatique : cron "0 * * * *" → toutes les heures pile
  Manuelle    : POST /sync/fabric (bouton dans Admin → Sync Fabric)
  La sync est "upsert" : elle ne supprime jamais les données locales.


7. PAGES ET FONCTIONNALITÉS DÉTAILLÉES
───────────────────────────────────────

PAGE D'ACCUEIL  /
  Tableau de bord avec cartes d'accès rapide.
  - Cartes : Planning Prévisionnel, Vue Gantt, Vue Chantier, Calendrier Personne
  - Section Administration : liens Ateliers, Personnel, Chantiers, Sync Fabric
  - Sélecteur de filiale active (contexte global)
  - Bouton de synchronisation Fabric avec statut en temps réel (spinner + résultats)

PAGE PLANNING  /planning
  Grille hebdomadaire principale. Cœur de l'application.
  - Header : sélecteur filiale + sélecteur atelier + navigation semaine
  - Navigation : semaine précédente / suivante / bouton "Aujourd'hui"
  - Colonnes : Lundi → Dimanche (7 jours). Week-end stylé différemment (bg-muted/20)
  - Lignes : une ligne par employé actif de l'atelier sélectionné
  - Cellules AM/PM : chaque jour est découpé en Matin (haut) et Après-midi (bas),
    chacun cliquable indépendamment
  - Clic sur AM ou PM → ouvre AssignmentModal
  - AssignmentModal : choix du chantier(s), OF associé, slot (AM/PM/Journée entière),
    temps passé (heures décimales), commentaire
  - Absences affichées visuellement dans la cellule (couleur + type)
  - Menu clic droit → options contextuelles (ajouter absence, dupliquer, etc.)
  - Sélection multiple de personnes via cases à cocher → barre d'actions groupées
  - Duplication de semaine : copier les affectations de la semaine précédente
  - Annuler / Rétablir (undo/redo) sur toutes les actions
  - Palette de commandes Ctrl+K : recherche rapide personne ou chantier
  - Journal d'activité : log des dernières actions effectuées
  - Panel de validation : mode "validation" activable pour soumettre des demandes
  - Export Excel de la semaine courante
  - Export PDF de la semaine courante
  - Affichage de densité : compacte / confort / spacieuse
  - KPIs en header : nombre d'employés, % jours remplis
  - État vide : si aucun atelier sélectionné → invite à sélectionner un atelier
  - État vide : si atelier sans membres → lien vers Gestion des Ateliers

PAGE GANTT  /gantt
  Vue Gantt des chantiers avec 3 couches visuelles superposées :
  1. Contractuel (baseline ERP) : barre hachurée / pointillée
  2. Prévisionnel (dates saisies manuellement) : barre claire
  3. Réel (calculé depuis les affectations enregistrées) : barre pleine + progression
  - Icônes de statut par chantier : A planifier / En cours / Bloqué / Terminé
  - Barres de progression visuelles (progressPct)
  - Navigation temporelle sur l'axe horizontal

PAGE VUE CHANTIER  /chantier
  Vue centrée sur un chantier / projet unique.
  - Panneau gauche : liste des projets avec leur statut et leur barre de progression
  - Panneau droit : grille hebdomadaire AM/PM pour le chantier sélectionné
  - Affiche les OFs (ordres de fabrication) liés au chantier
  - Liste des intervenants affectés à ce chantier + totaux temps passé
  - Navigation semaine comme la page planning principale

PAGE CALENDRIER PERSONNE  /person-calendar
  Calendrier individuel pour un employé.
  - Basé sur FullCalendar v6
  - Vues : mois, semaine
  - Affiche toutes les affectations et absences de la personne sélectionnée
  - Sélecteur d'employé dans le header

PAGE ADMIN — GESTION DES ATELIERS  /admin/teams
  Gestion des workshops (ateliers) groupés par filiale.
  - Toutes les filiales affichées, chacune expansible
  - Chaque filiale affiche ses ateliers avec le nombre de membres
  - Bouton "Nouvel atelier" par filiale → formulaire inline
  - Création d'un atelier : saisie du nom → POST /workshops
  - Renommer un atelier : édition inline → PATCH /workshops/:id
  - Supprimer un atelier → DELETE /workshops/:id
  - Expansion d'un atelier → liste des membres actuels
  - Champ de recherche "Affecter une personne" → dropdown autocomplete
  - Affecter un employé : PATCH /employees/:id { workshopId }
    (déplace l'employé d'un atelier à l'autre automatiquement)

PAGE ADMIN — GESTION DU PERSONNEL  /admin/people
  Gestion de tous les employés.
  - Liste complète avec code, nom, prénom, atelier, statut actif/inactif
  - Créer un nouvel employé → POST /employees
  - Modifier un employé (nom, prénom, code, atelier, statut) → PATCH /employees/:id
  - Archiver / Réactiver un employé (isActive)
  - Filtrage par atelier, recherche par nom/code
  - Sélecteur atelier pour chaque employé (workshopId)

PAGE ADMIN — GESTION DES CHANTIERS  /admin/projects
  CRUD complet des projets/chantiers.
  - Liste des chantiers avec code, label, statut, progression, dates
  - Créer un chantier → POST /projects
  - Modifier : code, label, couleur, dates contractuelles, dates prévisionnelles,
               statut, progression (%) → PATCH /projects/:id
  - Activer / Désactiver un chantier
  - Gérer les OFs (ordres de fabrication) associés
  - Supprimer un chantier → DELETE /projects/:id

PAGE ADMIN — FILIALES & ATELIERS  /admin/subsidiaries
  Gestion des entités organisationnelles.
  - CRUD des filiales : créer, modifier, supprimer
  - CRUD des ateliers par filiale (vue complémentaire à /admin/teams)

PAGE ADMIN — SYNC FABRIC  /admin/sync
  Interface de synchronisation ERP.
  - Bouton "Lancer la synchronisation" → POST /sync/fabric
  - Résultats affichés : nombre de filiales, ateliers, employés,
    chantiers, OFs synchronisés
  - Liste des erreurs éventuelles
  - Durée de la synchronisation affichée
  - Statut de la dernière synchronisation automatique


8. ÉTAT GLOBAL (PlanningStoreContext)
──────────────────────────────────────

11. POINTS IMPORTANTS À RETENIR
─────────────────────────────────

1. ATELIERS = WORKSHOPS (PAS les "teams" de la BDD)
   La table Team existe en BDD mais est non active dans l'UI actuelle.
   L'unité de planification est le Workshop. Employee.workshopId
   détermine qui apparaît dans quelle grille de planning.

2. SYNC FABRIC = UPSERT uniquement
   La sync ne supprime jamais de données locales.
   Elle crée si absent, met à jour si présent.
   Les données saisies manuellement (plannedStart, progressPct, etc.)
   ne sont jamais écrasées par la sync.

3. MODE OFFLINE
   Sans VITE_API_URL dans .env, toute l'app tourne sur mockData.ts.
   Utile pour démo ou développement frontend sans backend.

4. SLOTS AM / PM / FULL
   Un employé peut avoir AM + PM le même jour (2 assignments différents).
   Chaque slot est une entrée séparée en BDD.
   La cellule de la grille est découpée visuellement en haut (AM) et bas (PM).

5. COULEURS DES CHANTIERS
   Générées algorithmiquement depuis le code commande.
   Stables entre les synchronisations : même code → même couleur toujours.

6. PERFORMANCES SQLITE
   PRAGMA WAL activé au démarrage du serveur.
   busy_timeout=5000ms pour gérer les accès concurrents.
   Index sur (employeeId, date) et (projectId) pour les affectations.
