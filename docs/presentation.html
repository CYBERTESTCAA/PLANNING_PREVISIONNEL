Le portier de l'application

Nginx c'est comme le receptionniste d'un hotel. Vous arrivez, il vous dit : la salle de reunion c'est a droite, le restaurant a gauche.

Concretement : quand vous tapez l'adresse du site, c'est Nginx qui vous envoie la page web. Et quand le site a besoin de donnees, Nginx redirige vers le moteur (Fastify).

Ses responsabilites

  • Servir les pages web (HTML/CSS/JS)
  • Rediriger les appels /api vers Fastify
  • Chiffrer la connexion (HTTPS = cadenas)

  • Mettre en cache pour accelerer

🚀 Fastify 5 + Prisma 6 + SQLite

Backend API REST — Node.js 20

Fastify est un framework HTTP Node.js plus performant qu'Express (~2x). Il utilise un systeme de plugins et de schemas JSON pour la validation automatique des requetes.

Prisma est l'ORM qui genere un client type-safe a partir du schema schema.prisma. On ecrit prisma.plan.findMany() au lieu de requetes SQL manuelles. SQLite stocke tout dans un seul fichier .sqlite.

Stack backend complete

  • Fastify 5 : routing, plugins, hooks lifecycle
  • Prisma 6 : ORM type-safe + migrations
  • SQLite : SGBD embarque (fichier unique)
  • Zod : validation des body/params
  • @fastify/cors : Cross-Origin (dev)
  • @fastify/helmet : headers securite
  • @fastify/rate-limit : anti-abus
  • @fastify/multipart : upload fichiers
  • mssql : connexion SQL Server (Fabric)
  • jsonwebtoken + jwks-rsa : verification JWT
  • node-cron : taches planifiees (sync, backup)

Exemple : une route Fastify (api/src/routes/plans.ts)

Chaque fichier de route exporte une fonction qui enregistre ses endpoints sur l'instance Fastify

// api/src/routes/plans.ts (simplifie) import { FastifyInstance } from 'fastify'; import { PrismaClient } from '@prisma/client'; import { z } from 'zod'; const prisma = new PrismaClient(); const CreatePlanSchema = z.object({ projectId: z.number(), hk: z.string().optional(), number: z.number(), articleIds: z.array(z.number()).optional(), }); export default async function planRoutes(app: FastifyInstance) { // GET /plans?projectId=123 app.get('/plans', async (req, reply) => { const { projectId } = req.query as { projectId: string }; const plans = await prisma.plan.findMany({ where: { projectId: Number(projectId) }, include: { articles: true }, orderBy: { number: 'asc' }, }); return plans; }); // POST /plans — cree un plan avec ses articles app.post('/plans', async (req, reply) => { const body = CreatePlanSchema.parse(req.body); const plan = await prisma.plan.create({ data: { projectId: body.projectId, hk: body.hk ?? null, number: body.number, articles: body.articleIds ? { connect: body.articleIds.map(id => ({ id })) } : undefined, }, include: { articles: true }, }); reply.code(201); return plan; }); }

🔐 Azure AD (Entra ID) + MSAL

Authentification OAuth2 / OpenID Connect

L'appli utilise @azure/msal-react cote frontend pour le flow OAuth2 Authorization Code + PKCE. L'utilisateur se connecte via la popup Microsoft, le frontend recoit un access token JWT.

Cote backend, chaque requete API porte le JWT dans le header Authorization: Bearer <token>. Fastify verifie la signature via JWKS (cles publiques Azure) avec jwks-rsa + jsonwebtoken.

Architecture d'auth

  • Frontend : MSAL popup → access_token
  • API calls : header Authorization: Bearer
  • Backend : middleware de verification JWT
  • JWKS : cles publiques Microsoft pour verifier
  • Tenant : Azure AD de l'entreprise CAA
  • Scopes : api://<app-id>/access
  • Fabric : Service Principal (client_credentials)

La synchronisation automatique avec le systeme HK via Microsoft Fabric.

Toutes les heures, l'application va chercher les donnees a jour dans le systeme HK (via Microsoft Fabric). C'est automatique, aucune action necessaire.

🛠 Ateliers et equipes

👤 Employes

Noms, matricules, qualifications. Mis a jour automatiquement.

🏗 Chantiers

Codes, libelles, dates, statuts. Importes de HK.

📦 OF et articles

Ordres de fabrication et leurs articles associes.

💼 Clients et affaires

Informations commerciales importees du systeme.

Ce qui est cree localement (dans l'application)

✅ Donnees locales

  • Le planning (qui travaille ou, quand)
  • Les absences (CP, RTT, maladie)
  • Les questions et reponses
  • Les plans de fabrication
  • Les pieces jointes (photos, PDF)

🔒 Jamais ecrasees

Les donnees locales ne sont JAMAIS ecrasees par la synchronisation HK. La sync importe uniquement les donnees de reference (employes, chantiers, OF...). Votre planning et vos questions sont toujours en securite.

Comment la sync Fabric fonctionne techniquement

Le backend se connecte au SQL Endpoint Fabric via le protocole TDS (mssql) avec un Service Principal Azure

// api/src/services/fabricSync.ts (simplifie) import sql from 'mssql'; import { DefaultAzureCredential } from '@azure/identity'; async function syncFromFabric() { // 1. Auth via Service Principal (client_id + client_secret) const credential = new DefaultAzureCredential(); const token = await credential.getToken('https://database.windows.net/.default'); // 2. Connexion au SQL Endpoint Fabric const pool = await sql.connect({ server: process.env.FABRIC_SQL_ENDPOINT, // ex: xxxx.datawarehouse.fabric.microsoft.com database: process.env.FABRIC_DATABASE, authentication: { type: 'azure-active-directory-access-token', options: { token: token.token } }, }); // 3. Requetes SELECT sur les tables Fabric (warehouse) const employees = await pool.request().query('SELECT * FROM wh_employees'); const projects = await pool.request().query('SELECT * FROM wh_projects'); // 4. Upsert dans SQLite via Prisma (create or update) for (const emp of employees.recordset) { await prisma.employee.upsert({ where: { code: emp.code }, create: { code: emp.code, firstName: emp.firstName, lastName: emp.lastName }, update: { firstName: emp.firstName, lastName: emp.lastName }, }); } } // 5. Planifie via node-cron (toutes les heures) cron.schedule(process.env.FABRIC_SYNC_CRON || '0 * * * *', syncFromFabric);

Section 6

Comment on met a jour l'application ?

Le processus est entierement automatique.

🎉

Bonne nouvelle : vous n'avez rien a faire ! Quand le developpeur envoie ses modifications, le serveur se met a jour tout seul en quelques minutes.

Les 3 etapes de la mise a jour

1. Le developpeur envoie ses modifications

Il modifie le code sur son PC (avec Windsurf), puis envoie les changements sur GitHub avec git push. C'est comme sauvegarder un document sur Google Drive.

2. Le serveur detecte les changements

Toutes les 5 minutes, le serveur verifie GitHub. S'il detecte du nouveau code, il le telecharge automatiquement (git pull).

3. L'application se reconstruit et redemarre

Docker reconstruit les conteneurs avec le nouveau code et les redemarre. L'application est mise a jour ! Les utilisateurs n'ont qu'a rafraichir leur page (F5).

Deux methodes de mise a jour

✅ Automatique (recommandee)

  • Le dev fait git push sur GitHub
  • Le serveur detecte en 5 min max
  • Rebuild + redemarrage automatique
  • Aucune action cote serveur !

🔧 Manuelle (instantanee)

  • Se connecter au serveur en SSH
  • Taper : cd /opt/planning
  • Lancer : ./scripts/deploy.sh
  • Ou : docker compose up -d --build
💡

Les donnees ne sont jamais perdues lors d'une mise a jour. Le planning, les questions, les plans... tout est dans la base de donnees qui est preservee dans un volume Docker separe.

Le mecanisme d'auto-deploy (systemd timer)

Un timer systemd sur la VM execute le script de deploiement toutes les 5 minutes

# /etc/systemd/system/planning-update.timer [Unit] Description=Check for planning updates [Timer] OnBootSec=2min OnUnitActiveSec=5min # toutes les 5 min Persistent=true [Install] WantedBy=timers.target
#!/bin/bash — scripts/deploy.sh (simplifie) cd /opt/planning # Verifie s'il y a du nouveau sur GitHub git fetch origin main LOCAL=$(git rev-parse HEAD) REMOTE=$(git rev-parse origin/main) if [ "$LOCAL" != "$REMOTE" ]; then echo "Nouveau code detecte, deploiement..." git pull origin main docker compose build --no-cache docker compose down docker compose up -d docker image prune -f # nettoie echo "Deploy OK a $(date)" >> /var/log/planning-deploy.log fi

La securite

Comment l'application protege vos donnees et vos acces.

🔐

Connexion securisee

  • Compte Microsoft professionnel (@caa)
  • Meme identifiant qu'Outlook et Teams
  • Pas de mot de passe supplementaire
  • Seuls les employes CAA autorises
🔒

Protection des donnees

  • Connexion chiffree HTTPS
  • Headers de securite (CSP, HSTS...)
  • Protection anti-abus (rate limiting)
  • Secrets dans un fichier protege (.env)
💾

Sauvegardes

  • Sauvegarde auto chaque nuit a 2h
  • 30 jours de sauvegardes conservees
  • Stockage sur le serveur
  • Restauration possible en 5 minutes
🌐

Reseau interne uniquement : l'application n'est accessible que depuis le reseau interne de CAA. Elle n'est pas visible depuis Internet, ce qui ajoute une couche de securite supplementaire.

Commandes utiles sur le serveur (via SSH)

docker compose ps
Affiche si les 2 conteneurs (web + api) tournent bien. Vous devriez voir "Up" pour les deux.
docker compose logs -f api — Logs du backend (requetes, erreurs, sync Fabric)
docker compose logs -f web — Logs du frontend/Nginx (acces, erreurs)
Appuyez sur Ctrl+C pour arreter.
docker compose restart — Redemarre les 2 conteneurs
docker compose down — Arrete tout
docker compose up -d — Relance tout en arriere-plan
cd /opt/planning
./scripts/deploy.sh
Ou manuellement : git pull && docker compose up -d --build
./scripts/backup.sh
Cree une copie de la base dans /var/backups/planning/

Pour le developpeur : travailler sur le projet

💻 Premiere installation

  1. Installer Node.js 20, Git, Windsurf (ou VS Code)
  2. Cloner le projet : git clone [URL GitHub]
  3. Installer les dependances : npm install
  4. Installer les dependances API : cd api && npm install
  5. Copier le fichier api/.env.example en api/.env
  6. Lancer le frontend : npm run dev
  7. Lancer le backend : cd api && npm run dev

🚀 Workflow quotidien

  1. Ouvrir le projet dans Windsurf
  2. Lancer le dev : npm run dev + cd api && npm run dev
  3. Modifier le code (Cascade peut aider !)
  4. Tester dans le navigateur (localhost:8080)
  5. Sauvegarder : git add .
  6. Commiter : git commit -m "description"
  7. Envoyer : git push (mise a jour auto en 5 min)

Exporter un chat Windsurf

💬

Pourquoi ? Pour qu'un autre developpeur comprenne ce qui a ete fait et puisse continuer. L'historique Cascade contient tout le contexte du projet.

Exporter

  1. Ouvrir Windsurf avec le projet
  2. Dans le panneau Cascade, cliquer sur ... en haut
  3. Selectionner Export Chat
  4. Sauvegarder le fichier .md genere
  5. Le partager avec le nouveau dev

Importer

  1. Le nouveau dev clone le repo GitHub
  2. Ouvre le projet dans Windsurf
  3. Colle le contenu exporte dans un nouveau chat
  4. Cascade comprend le contexte et peut continuer
  5. Alternative : le Checkpoint se charge automatiquement
